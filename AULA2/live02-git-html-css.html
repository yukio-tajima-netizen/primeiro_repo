Live 02 · Versionamento & Front-end

Por que versionar desde o primeiro arquivo?

Você já viu uma pasta de projeto assim? Todo programador tem uma história parecida.

📁 meu-projeto/
  📄 index.html
  📄 index_backup.html
  📄 index_v2.html
  📄 index_v2_FINAL.html
  📄 index_v2_FINAL_corrigido.html
  📄 index_v2_FINAL_corrigido2.html
  📄 index_v2_FINAL_USE_THIS_ONE.html
  📄 index_PARA_APAGAR.html
  📄 style_bom.css
  📄 style_ANTIGO.css
Git resolve exatamente isso. Com controle de versão, você tem um único arquivo com um histórico completo de todas as versões — cada uma com data, hora, autor e uma mensagem explicando o que mudou.

O que é uma IDE?

🧰

IDE — Integrated Development Environment

Uma IDE é um editor de código turbinado: além de escrever texto, ela entende a linguagem que você está usando, sugere completamentos, destaca erros, integra com o terminal, o Git e dezenas de outras ferramentas — tudo em uma janela só.

Pense nela como a diferença entre um bloco de notas e o IntelliJ ou Eclipse que você já usou para Java — a IDE te ajuda a trabalhar, não só a escrever.

informativo

informativo

Fundamentos do Terminal

Linux: a língua dos servidores

Mais de 96% dos servidores web no mundo rodam Linux. O GitHub Actions, a maioria dos ambientes de desenvolvimento e os contêineres Docker são todos Linux. Saber se virar no terminal não é opcional — é uma habilidade fundamental.

🖥️

Terminal vs GUI

A interface gráfica (GUI) é uma tradução visual dos comandos. O terminal acessa o sistema diretamente — mais rápido, mais poderoso, scriptável.

📂

Tudo é arquivo

No Linux, tudo — discos, dispositivos, processos — é representado como arquivo. Entender o sistema de arquivos é a base de tudo.

Controle de Versão

Git — como ele pensa sobre seu código

Git não salva "diferenças entre arquivos" como muitas pessoas imaginam. Ele tira fotografias do estado completo do seu projeto em um momento específico. Cada commit é uma dessas fotos.

Sistema Distribuído

Cada desenvolvedor tem uma cópia completa do histórico. Não depende de servidor central para funcionar.

Histórico imutável

Uma vez commitado, nada some. Você pode sempre voltar a qualquer ponto no tempo.

Orientado a conteúdo

Git identifica arquivos pelo conteúdo (hash SHA-1), não pelo nome. Renomear não "perde" histórico.

Os 3 estados do Git

Esta é a ideia mais importante do Git. Todo arquivo no seu projeto está sempre em um desses três estados:

← git restore

📦 Preparado
→ git commit
← git reset

Primeiro repositório — passo a passo

① Configurando o Git pela primeira vez +

Antes de tudo, diga ao Git quem você é. Isso vai aparecer em cada commit:

# Configuração global (feita uma vez só) git config --global user.name "Seu Nome" git config --global user.email "seuemail@provedor.com" git config --global init.defaultBranch main # Verificar as configurações git config --list
② Iniciando um repositório +
# Criar pasta e entrar nela mkdir portfolio-card cd portfolio-card # Inicializar o Git nesta pasta git init Initialized empty Git repository in /portfolio-card/.git/ # Renomear branch principal para 'main' git branch -M main
Isso cria uma pasta oculta .git/ onde o Git armazena todo o histórico do projeto. Apagar essa pasta = apagar todo o histórico.
③ Verificando o status — git status +
git status On branch main No commits yet Untracked files: (use "git add <file>..." to include in what will be committed) index.html style.css nothing added to commit but untracked files present

Arquivos vermelhos = não estão sendo rastreados ou foram modificados. Use git status constantemente — ele explica o que fazer a seguir.

④ Preparando arquivos — git add +
# Adicionar um arquivo específico git add index.html # Adicionar todos os arquivos da pasta atual git add . # Verificar o que está pronto para commit git status Changes to be committed: new file: index.html new file: style.css
A Staging Area existe para você ter controle fino. Você pode commitar alguns arquivos mas não outros — por exemplo, commitar o CSS mas deixar o HTML ainda em desenvolvimento.
⑤ Criando uma versão — git commit +
git commit -m "feat: cria estrutura inicial do cartão de portfólio" [main (root-commit) a3f9c12] feat: cria estrutura inicial do cartão de portfólio 2 files changed, 45 insertions(+) create mode 100644 index.html create mode 100644 style.css
Boas mensagens de commit importam. Use verbos no imperativo, em inglês ou português. Ex: feat: adiciona seção de habilidades, fix: corrige layout mobile, style: ajusta paleta de cores.
⑥ Consultando o histórico — git log +
# Ver histórico completo git log # Versão compacta (a mais usada no dia a dia) git log --oneline a3f9c12 (HEAD -> main) feat: adiciona seção de habilidades d2e7b1a feat: cria estrutura inicial do cartão de portfólio # O hash (a3f9c12) identifica cada commit de forma única

Voltando no tempo — git reset

Uma das superpoderes do Git é poder voltar a qualquer ponto do histórico.

ComandoHEADStagingArquivosUso
git reset --soft <hash>voltamantémmantémRefazer mensagem do commit
git reset --mixed <hash>voltalimpamantémReorganizar o que vai no commit
git reset --hard <hash>voltalimpaapaga⚠️ Descartar tudo — sem volta
Front-end para Programadores

HTML & CSS — traduzindo o que você já sabe

Se você já programa, você já entende estruturas de dados, hierarquia e separação de responsabilidades. HTML e CSS são só um vocabulário novo para conceitos que você já conhece.

📄

HTML

O esqueleto. Define a estrutura e o significado do conteúdo. Pense como um JSON: hierarquia de elementos, com atributos.

🎨

CSS

A aparência. Define como os elementos são exibidos. Pense como uma stylesheet de configuração: propriedade: valor.

⚡

JavaScript

O comportamento. Torna a página dinâmica. É o que você aprenderá depois — foca no HTML+CSS por agora.

A estrutura de um documento HTML

Todo arquivo HTML tem a mesma estrutura básica — como a estrutura de uma classe em qualquer linguagem:

📄 index.html — estrutura mínima
<!DOCTYPE html> ← declara o tipo de documento <html lang="pt-BR"> ← raiz do documento <head> ← metadados (não aparece na tela) <meta charset="UTF-8"/> <meta name="viewport" content="width=device-width"/> <title>Meu Portfólio</title> <link rel="stylesheet" href="style.css"/> </head> <body> ← conteúdo visível <h1>Olá, mundo!</h1> <p>Meu primeiro parágrafo.</p> </body> </html>

Tags essenciais — o vocabulário básico

Vamos focar nas tags de conteúdo — as que você vai usar para montar o cartão de portfólio desta live. Tags semânticas estruturais (<header>, <nav>, <section>, etc.) e multimídia são tema da próxima live.

Contêineres

<main>Área principal da página

Conteúdo

Tag de abertura e fechamento: A maioria das tags tem par — <p>texto</p>. Algumas são void elements e ficam sozinhas: <img />, <br />, <input />. O atributo alt na imagem é obrigatório — descreve a imagem para leitores de tela e SEO.

CSS — seletores e propriedades

A sintaxe do CSS é simples: quem você quer estilizar e o que você quer mudar.

🎨 style.css
/* Seleciona por tag */ body { font-family: 'Segoe UI', sans-serif; background-color: #f0f4f8; margin: 0; } /* Seleciona por classe (.nome) */ .card { background: #fff; border-radius: 12px; padding: 24px; box-shadow: 0 4px 12px rgba(0,0,0,.1); } /* Seleciona por id (#nome) */ #nome { font-size: 2rem; color: #1a1a2e; }
📄 index.html (usando as classes)
<body> <div class="card"> <h1 id="nome">Ana Silva</h1> <p>Desenvolvedora Full-Stack</p> </div> </body> /* Analogia com POO: tag → tipo (classe Java) classe → interface/trait id → instância específica .card aplica estilos a TODOS os elementos com class="card" #nome aplica APENAS ao elemento com id="nome" */

O Box Model — entendendo o espaçamento

Todo elemento HTML é uma caixa. Entender o box model é entender por que as coisas ficam onde ficam:

Conteúdo
Passe o mouse nas camadas →
margin — espaço fora da borda. Separa elementos entre si. Não tem cor de fundo.
border — a borda visível. Pode ter espessura, estilo e cor.
padding — espaço dentro da borda. Tem cor de fundo. Afasta o conteúdo das bordas.
content — o conteúdo real: texto, imagem, etc. width e height controlam este tamanho.
Dica: Use box-sizing: border-box para que width inclua padding e border. Muito mais previsível!
Na próxima live: vamos aprofundar HTML com semântica, metatags e multimídia (imagem, vídeo, áudio e iframe). No CSS, entraremos em Flexbox, Grid e media queries para criar layouts responsivos. Por enquanto, concentre-se em dominar as propriedades básicas e o box model!
Colaboração e Repositório Remoto

GitHub — o Git na nuvem

Git é a ferramenta local. GitHub é uma plataforma que hospeda repositórios Git remotamente, adicionando uma camada de colaboração: pull requests, issues, Actions, Pages, e muito mais.

🐙

git ≠ GitHub

git é o software de controle de versão (open source, criado por Linus Torvalds). GitHub é uma empresa (da Microsoft) que oferece hospedagem de repos Git. Existem alternativas: GitLab, Bitbucket.

🔗

origin

Quando você conecta seu repo local a um remoto, o Git chama esse remoto de origin por convenção. Você pode ter múltiplos remotos com nomes diferentes.

Conectando local ao remoto

# Após criar o repo no GitHub, conectar o local: git remote add origin https://github.com/seu-usuario/portfolio-card.git # Verificar a conexão git remote -v origin https://github.com/seu-usuario/portfolio-card.git (fetch) origin https://github.com/seu-usuario/portfolio-card.git (push) # Enviar para o remoto pela primeira vez git push -u origin main Branch 'main' set up to track remote branch 'main' from 'origin'.

Branches — trabalhando em paralelo

Uma branch é uma linha independente de desenvolvimento. Pense como um fork do histórico: você experimenta em segurança sem afetar a versão principal.

main     ● ──────────────────── ● ────── ●  ← merge commit
╲                        ╱
● ────── ●   ← branch "skills"

Comandos de branch

# Criar uma nova branch e mudar para ela git branch skills git checkout skills # ou em um único comando: git checkout -b skills Switched to a new branch 'skills' # Ver todas as branches git branch -a # Subir a nova branch para o GitHub git push origin skills # Voltar para main e fazer merge git checkout main git merge skills git push origin main

push, pull e fetch — a diferença importa

ComandoDireçãoO que fazQuando usar
git push origin main local → remoto Envia seus commits para o GitHub Depois de commitar, para compartilhar
git pull origin main remoto → local Baixa E integra as mudanças (fetch + merge) Para se atualizar com o trabalho do time
git fetch remoto → local repo Baixa sem integrar (você decide como mesclar) Para ver o que mudou sem alterar seu trabalho
git clone <url> remoto → local Copia o repo inteiro para sua máquina Primeira vez que você pega um repo
Conflito de merge: Quando dois commits alteram a mesma linha do mesmo arquivo, o Git não sabe qual versão manter. Ele marca o conflito no arquivo com <<<<<<< e >>>>>>>. Você resolve manualmente, salva, faz git add e git commit.
Deploy gratuito

GitHub Pages — sua página no ar em 2 minutos

GitHub Pages é um serviço gratuito do GitHub que publica seu repositório como um site estático. Perfeito para portfólios, documentações e projetos front-end.

🆓

100% gratuito

Para repositórios públicos, sem custo algum. Ótimo para projetos pessoais e acadêmicos.

🔗

URL automática

Você ganha uma URL no formato: usuario.github.io/repositorio

🔄

Deploy automático

Cada push para a branch configurada atualiza o site em poucos segundos.

Ativando o GitHub Pages

1

Crie o repositório e faça o primeiro push

Seu repositório precisa ter pelo menos um commit e o arquivo index.html na raiz da branch que você vai publicar.

git push origin main
2

Acesse as configurações do repositório

No GitHub, abra seu repositório → clique em Settings (ícone de engrenagem no topo).

3

Ative o Pages

No menu lateral esquerdo, clique em Pages. Em Branch, selecione main e a pasta / (root). Clique em Save.

4

Aguarde o deploy (≈ 1 minuto)

Aguarde cerca de 1 minuto. Recarregue a página de Settings → Pages. Vai aparecer uma caixa verde com sua URL:

🎉 Your site is live at https://seu-usuario.github.io/portfolio-card/
5

Atualizações automáticas

A partir de agora, cada git push origin main dispara um deploy automático. Você verá uma bolinha amarela (processando) ou verde (publicado) no seu repositório.

# Edite, commite e veja ao vivo git add . git commit -m "style: atualiza cores do cartão" git push origin main # Em ~30s, o site é atualizado automaticamente!
Limitações do GitHub Pages: Apenas sites estáticos (HTML, CSS, JS). Não executa back-end (Python, Java, Node.js com banco de dados). Para projetos full-stack, você precisará de Heroku, Vercel, Render ou similares.

Desafio Final — ao vivo

Agora você vai sentir a magia do deploy contínuo. Uma mudança no código, um commit bem descrito, e em menos de 1 minuto o mundo pode ver.

A regra: Qualquer alteração vale. Mude uma cor, adicione uma habilidade, escreva uma frase diferente na bio. O que importa é o commit com mensagem descritiva e ver a mudança publicada em tempo real.

Execute agora:

# 1. Faça qualquer alteração no index.html ou style.css # (pode ser mudar uma cor, adicionar uma habilidade, etc.) # 2. Verifique o que mudou git status git diff # 3. Prepare e commite com uma mensagem DESCRITIVA git add . git commit -m "style: muda cor do fundo para azul-marinho" # 4. Publique git push origin main # 5. Aguarde ~30s e recarregue o seu site no GitHub Pages # 🎉 Sua alteração está ao vivo!
